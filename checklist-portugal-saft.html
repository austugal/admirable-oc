Section 01 · Free

Property & Country Mode setup

Country Mode is the master switch that activates Portugal-specific fiscal logic across the entire OPERA Cloud property. Without it, none of the PT-specific exports, folio types, or AT integrations become visible. This must be set before any other configuration begins.

Configuration checks

  • Country Mode set to PT at Administration → Enterprise → Properties → [Property] → General Information. Verify by opening any new reservation: PT-specific fields (NIF, Cidadão Portuguese ID checkbox) must appear.
  • Default currency set to EUR. Multi-currency posting allowed but EUR must be the property base currency for SAF-T compliance.
  • Property timezone set to Europe/Lisbon (UTC+0/+1). Critical for Business Date alignment with AT submission deadlines.
  • Property language includes Portuguese (PT). Folios printed for guests must offer PT as a language option even if EN is the staff working language.
  • Fiscal Folio Printing OPERA Control activated under Administration → Enterprise → OPERA Controls → Cashiering group. This unlocks the fiscal folio workflow used by every charge posting downstream.

Common mistakes

Setting Country Mode after going live causes existing reservations and folios to render in the wrong format. Country Mode must be set during Configuration, before any production data exists. If switching mid-life, all existing folios become non-compliant for AT and require manual credit-note + re-issue workflow.

Section 02 · Free

Document numbering & series

Portuguese law requires sequential, gap-free numbering for every fiscal document type independently. FT, FS, NC, ND, FR each have their own series. Once a number is allocated it cannot be reused, even if the document is voided. AT will reject SAF-T exports with sequence gaps.

Configuration checks

  • Each fiscal document type (FT, FS, NC, ND, FR) has at least one series defined under Administration → Financial → Cashiering Management → Document Types. Series codes typically prefixed by year (FT2026) or property (FT-LIS-26).
  • Starting number set per series. Standard practice: start at 1 for new property, continue sequence for existing property migrating from another PMS. Document the carry-over with the AT before go-live.
  • Series validity dates set with end date in the future. Most properties define series annually with a 31 December cutover and 1 January start of new series.
  • Communication to AT completed for each new series via Portal das Finanças before first use. AT issues a series validation code that must be entered in OPERA against the series record.
  • No deletion permission for fiscal document records confirmed at the user role level. Only management roles should have void/cancel permission, never delete. Deletion of a fiscal document creates an unrecoverable gap.

Common mistakes

Allowing front-desk roles to void documents without supervisor approval. Voids appear as gaps in the SAF-T sequence and must be paired with credit notes referencing the voided document. The standard workflow is: post FT → guest disputes → issue NC referencing FT → both appear in SAF-T, sequence intact.

Section 03 · Free

Fiscal folio types & print tasks

OPERA Cloud uses folio templates to generate the visible PDF/print output. Portugal-specific templates include the digital signature hash, ATCUD code, QR code, and required legal text. Using a generic template produces a non-compliant invoice that AT will reject.

Configuration checks

  • sample_folio_portugal stationery template imported and active. Customised version saved under property-specific name (folio_lis_pt) so future Oracle updates don't overwrite.
  • SIGNATURE_HASH and SIGNATURE_HASH_FOOTER merge codes present in the folio template. These render the digital signature required for AT compliance. Missing merge codes mean printed folios are not legally valid.
  • ATCUD code merge field present. This is the unique document code generated from the AT-issued series validation code plus the document number.
  • QR code rendering correctly on the folio. Test by scanning printed output with any QR reader. The decoded string must contain document type, NIF, gross total, and signature hash per AT specification.
  • Folio Type Prefix and Suffix OPERA Control activated where the property uses prefixed numbering (FT2026/000123). Validate that the prefix appears in both the printed folio AND the SAF-T export.
  • Folio No Reprint OPERA Control activated. Required as a prerequisite for the Print Fiscal Information Folio control. Without this, reprints generate new fiscal numbers, breaking sequence.

Common mistakes

Going live with the generic sample_folio.rtf template instead of sample_folio_portugal.rtf. The generic template is missing all PT-specific merge codes. First invoices print without signature, ATCUD, or QR. Properties typically discover this on day two when AT rejects the first SAF-T submission.
